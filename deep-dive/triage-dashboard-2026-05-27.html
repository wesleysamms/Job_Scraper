1. Architecture: fetch-at-view-time

The dashboard could have been built two fundamentally different ways. We picked one. Here's the choice space:

Option A — bake-on-cron rejected

Every cron fire (LinkedIn :17, Indeed :47, biotech daily) regenerates triage.html with the latest data inlined as a JSON blob in a <script> tag. Commit the rebuilt HTML on every push.

  • + Works offline (open the file directly)
  • – ~30 commits/day just for the dashboard
  • – Couples Python scraper to dashboard format

Option B — fetch-at-view-time chosen

The HTML is static. When the browser loads it, JavaScript runs fetch('./jobs.json'), fetch('./linkedin_jobs.json'), fetch('./indeed_jobs.json') in parallel from the same origin, merges them, and renders.

  • + HTML committed once, never modified by automation
  • + Decouples dashboard from scraper
  • – Requires HTTP (won't work from file://)

Why this is the right call for personal projects

The pivot point is whether you have a server. If you don't (and you don't — Pages is static file hosting, not a runtime), the data has to live somewhere. Either you compile-time-inline it (Option A) or you fetch-time-load it (Option B). Option B is cheaper because the data is already in the repo as JSON; you just point at it.

This pattern is common in static-site dashboards: Observable notebooks, D3 demos, Datasette, and most "single-page apps backed by static files" use it. You're now in good company.

2. Promise.allSettled — fail-soft parallel fetching

The code:

const settled = await Promise.allSettled(
  SOURCES.map(s => fetch(s.name + bust).then(r => {
    if (!r.ok) throw new Error(`${s.name}: ${r.status}`);
    return r.json();
  }))
);

What Promise.allSettled takes an array of promises and waits for all of them to either resolve or reject. It returns an array of result objects: {status: 'fulfilled', value: ...} or {status: 'rejected', reason: ...}. No matter what any individual promise does, the outer promise never rejects.

Why we chose it The 3 sources are independent. If indeed_jobs.json doesn't exist yet (first deploy before the Indeed cron has fired), fetch() returns a 404 and our r.ok check throws. With Promise.all, that throw would reject the whole batch — you'd see zero jobs even though biotech + LinkedIn loaded fine. With Promise.allSettled, the 404 is contained: the per-source loop just skips it.

When to reach for it Any time you're fanning out independent requests and partial success is acceptable. Common cases: dashboards aggregating multiple APIs, batch operations, anything where "best effort" beats "all or nothing".

Python analogue: asyncio.gather(*tasks, return_exceptions=True). Same semantics — return both results and exceptions, you decide what to do with each. Without return_exceptions=True, gather aborts the whole batch on the first exception, just like Promise.all.

Why not Promise.all?

APIBehavior on failureReturns
Promise.allRejects whole batch on first rejectionArray of values, or throws
Promise.allSettledAlways resolves, never rejectsArray of {status, value/reason}
Promise.raceResolves/rejects with first to finishSingle value
Promise.anyResolves with first success; rejects only if all failSingle value, or AggregateError

The decision tree: need all results, OK if some fail → allSettled. Need all results, abort on any failure → all. Want fastest one → race. Want any one to succeed → any.

3. localStorage + additive merge pattern

localStorage is a browser-provided key/value store that persists across page loads. It's synchronous (no async API), strings-only (so we JSON-stringify objects), and ~5MB per origin.

const state = (() => {
  try { return JSON.parse(localStorage.getItem(STORAGE_KEY)) || {}; }
  catch { return {}; }
})();
state.triage = state.triage || {};
state.jobs = state.jobs || [];

What An immediately-invoked function expression (IIFE) reads the saved state on page load. If the parse fails (corrupted storage, first visit), we fall back to an empty object. Defensive coding for browser storage — never assume your data is intact.

Why JSON.parse with try/catch Browser storage can be corrupted by extensions, manually edited, or in private/incognito mode where some browsers silently fail. The try/catch is cheaper than the alternative (handling broken state mid-render).

The additive-merge pattern

The harder design question: when fresh data arrives, do you replace the in-memory jobs list, or merge into it?

function mergeJobs(incoming) {
  const seen = new Map(state.jobs.map(j => [j.url, j]));
  let added = 0;
  for (const j of incoming) {
    if (!seen.has(j.url)) {
      state.jobs.push(j);
      seen.set(j.url, j);   // prevent internal dupes within same call
      added++;
    }
  }
  return added;
}

Why additive, not replace?

Each scraper has a small lookback window (LinkedIn: 1h, Indeed: 1h, biotech: 24h). If you replace state.jobs with each fresh fetch, jobs you triaged yesterday vanish from the visible list — but your state.triage[url] = 'applied' decision still exists in localStorage, just orphaned. Your "Applied: 5" KPI would silently drop to "Applied: 0" overnight.

Additive merge keeps every job you've ever seen. The lookback windows give you fresh data while your historical decisions stay anchored. The cost is unbounded growth — but at 39 jobs per hour and ~200 bytes each, you'd hit 5MB after about 4 months. We'll cross that bridge if/when we get there.

localStorage vs alternatives

StorageCapacityAPIUse case
localStorage~5MBSync, string-onlySmall JSON blobs, settings, simple state
sessionStorage~5MBSync, string-onlyTab-lifetime only — cleared on close
IndexedDB~50% of diskAsync, structuredLarge datasets, offline-first apps
Cookies~4KB totalSent on every HTTP requestAuth tokens, server-side session
Cache APIOrigin-quotaAsync, request/responseService workers, offline assets

For our scale (a few KB of triage state), localStorage is right. For a Kaggle-sized dataset or anything you want offline-first, IndexedDB is the move — but it has a steep API.

4. Map keyed by URL — why not plain object?

const byUrl = new Map();
// ...
if (!byUrl.has(j.url)) byUrl.set(j.url, j);
// ...
const jobs = [...byUrl.values()];

What ES6 Map is a key/value container where keys can be any type (not just strings) and iteration order matches insertion order. has, set, get, delete, values(), keys(), entries(), size — all the standard collection ops.

Why over {} Three reasons: (1) explicit semantics — Map is clearly a collection, an object isn't always; (2) no prototype-key collisions (object keys conflict with toString, hasOwnProperty, etc.); (3) .size is O(1) where Object.keys(o).length is O(n).

When Any time the keys are dynamic strings (especially URLs, IDs, or user-supplied values). Use plain objects only when the shape is fixed and known at write-time.

Python analogue: dict. JavaScript objects were dual-purpose (record + dict) for years; ES6 split them into proper Object (record) and Map (dict). Modern JS strongly prefers Map for dictionary use cases.

5. Cache-busting via ?v=Date.now()

const bust = `?v=${Date.now()}`;
fetch(s.name + bust);   // /jobs.json?v=1748391982371

What Appending a unique query string to a URL makes it look "new" to caches (browser, CDN, ISP). Same content, but the cache key changes per request, so a fresh fetch is forced.

Why we need it GitHub Pages serves static files with Cache-Control: max-age=600 — the browser will reuse the cached copy for 10 minutes before re-checking. After a cron commits new JSON, you'd see stale data for up to 10 min. The query bust skips browser cache; whether it skips the upstream CDN depends on the CDN's cache-key rules (Fastly typically respects query strings).

When Static-hosted dynamic data (Pages, S3, Netlify with default cache). Avoid for hashed-build assets like app.a1b2c3.js — those should be cached aggressively because the hash is the bust.

Better alternatives if you control the server

  • ETag + If-None-Match: server says "still the same, here's a 304" without re-sending the file. Standard HTTP, no client work needed.
  • Cache-Control: no-cache on the server: force revalidation every request. Cleaner than query bust but requires server config.
  • Versioned URLs (/v2/jobs.json): when content changes, you change the path. Works great for build pipelines.

6. Defensive coding: HTML escape + URL protocol guard

Two layered defenses against malicious or malformed data.

HTML escape

function escape(s) {
  return String(s == null ? '' : s).replace(/[&<>"']/g, c =>
    ({ '&': '&amp;', '<': '&lt;', '>': '&gt;',
       '"': '&quot;', "'": '&#39;' }[c]));
}

Any time data goes into .innerHTML or a template literal that gets injected as HTML, it must be escaped. A job title containing <script>alert(1)</script> would otherwise execute as JavaScript. Five chars are dangerous: & < > " '. The replace function maps each to its named/numeric entity.

Production note: if you find yourself writing escape functions, consider using .textContent (auto-escapes) or a templating library (Lit, Preact, etc.) that escapes by default. We use .innerHTML here for batched template literal rendering performance with small DOM nodes — a deliberate trade-off.

URL protocol whitelist

(d.jobs || []).forEach(j => {
  if (!j || !j.url || !/^https?:\/\//i.test(j.url)) return;
  if (!byUrl.has(j.url)) byUrl.set(j.url, j);
});

The regex /^https?:\/\//i requires URLs to start with http:// or https:// (case-insensitive). A malicious or buggy scraper that produced javascript:alert(1) would otherwise become a clickable link in the UI — tap it on iPhone, code executes. The guard prevents that whole class of attack.

Real-world incidence: very low for personal scrapers hitting LinkedIn/Indeed/Greenhouse. But the guard costs one regex test and prevents a footgun. Cheap insurance.

7. Event delegation

$('jobs').addEventListener('click', e => {
  const btn = e.target.closest('.act');
  if (!btn) return;
  const card = btn.closest('.job');
  const url = card.dataset.url;
  const act = btn.dataset.act;
  // toggle triage state
});

What One click listener on the parent #jobs container catches clicks from any descendant. We use e.target.closest('.act') to find the nearest matching ancestor (or self) of the actual clicked element. If null, the click wasn't on a triage button — bail out.

Why Each render produces N jobs × 3 buttons = up to ~120 buttons. Attaching 120 individual listeners costs memory and slows DOM mutations. One listener handles them all. More importantly, jobs render multiple times (filter changes, refreshes), and you'd otherwise need to wire/unwire listeners constantly. Event delegation works automatically for elements created after the listener was attached.

When Anytime you have dynamic lists with interactive items. Tables, cards, list rows, autocomplete dropdowns — all benefit. closest() is the key API: it lets you write the listener once and match by class/tag regardless of nesting.

How events propagate (so this works)

When you click an element, the browser runs a bubble phase: the event fires on the target, then on its parent, then grandparent, all the way up to document. Each ancestor's listeners get called in order. Our listener on #jobs sees clicks bubbled up from any descendant button. Add e.stopPropagation() if you ever need to prevent this — but usually you want it.

8. CSS custom properties for theming

:root {
  --bg: #fafaf7;
  --ink: #1a1a1a;
  --accent: #d94f1e;
  /* ... */
}
[data-theme="dark"] {
  --bg: #131310;
  --ink: #ece9e2;
  /* ... */
}
body { background: var(--bg); color: var(--ink); }

What CSS variables (formal name: "custom properties") declared on :root cascade everywhere. Override them at any nested scope and descendants get the new value. var(--name) reads the closest declaration.

Why for theming You define every theme-able value once at :root, then a single class/attribute change at the top of the tree flips the whole UI. No re-running JS, no rebuilding the DOM, no SCSS mixins compiled per-theme. The dark mode toggle is literally html.dataset.theme = 'dark' — one assignment, the entire palette swaps.

When Theming, spacing scales, density modes, brand variants, anywhere you want "change one thing, everything updates". CSS variables compose with media queries (prefers-color-scheme), data attributes, and JS, so you can wire them however you need.

System preference + manual override

applyTheme(localStorage.getItem('jobTriage:theme') ||
  (window.matchMedia('(prefers-color-scheme: dark)').matches ? 'dark' : 'light'));

The user's saved choice wins; falling back to the system preference if they've never toggled. matchMedia returns a MediaQueryList you can also subscribe to (for live updates if the OS changes its theme), though we don't bother here.

9. Mobile gotchas — the iOS WebKit reality

The one thing to internalize

iOS Chrome is not Chrome. Apple App Store policy requires every iOS browser to use WebKit (Safari's engine) for HTML/CSS/JS rendering. Chrome, Firefox, Edge on iPhone share the WebKit reality. So when we say "test on iPhone Chrome", we're really testing on Safari with a Chrome wrapper.

Every quirk below is a WebKit quirk that affects all iOS browsers equally.

iOS auto-zoom on input focus (the most famous one)

@media (max-width: 760px) {
  .search, .modal textarea { font-size: 16px; }
}

Any <input> or <textarea> with font-size less than 16px triggers iOS to auto-zoom the page when the user taps to focus. The page stays zoomed in until they manually pinch-zoom out. Bumping to 16px (or larger) on mobile prevents this. Memorize this rule. You'll hit it in every mobile project.

Touch targets — Apple HIG 44pt minimum

.act { padding: 10px 14px; min-height: 44px; }

Apple's Human Interface Guidelines mandate 44×44pt minimum for interactive elements. Anything smaller is realistically un-tappable with a finger — mishits become the norm. The 44pt comes from research on the average fingertip contact area (~10mm) and screen pixel density.

Safe area insets

.export-bar {
  padding-bottom: calc(10px + env(safe-area-inset-bottom));
}

iPhones with home indicators or notches expose CSS environment variables: env(safe-area-inset-top/right/bottom/left). These are the pixel distances you should respect to avoid your content hiding behind the notch/indicator. viewport-fit=cover in the meta tag opts in to using the full screen, and these env vars let you reclaim usable space.

position: sticky vs fixed

WebKit has historic bugs with position: sticky inside scrolling containers, and it interacts badly with iOS Chrome's auto-hiding bottom toolbar. We use sticky on desktop (cheaper, scrolls with content) but switch to position: fixed on mobile for the export-bar specifically. Worth knowing if you ever debug "why doesn't my sticky element stick on iPhone".

-webkit-tap-highlight-color: transparent

iOS draws a translucent gray rectangle over any clickable element when you tap it. Looks crude on a custom-styled UI. Setting -webkit-tap-highlight-color: transparent removes it. Your CSS hover/active states should provide tap feedback instead.

10. Accessibility primitives

The dashboard had several a11y bugs in the v1 prototype that pass 7 of the audit caught. Each fix maps to a learnable concept.

Real <button> vs <span onclick>

Filter pills were originally <span class="pill"> with onclick handlers. They worked with mouse and touch but were invisible to:

  • Keyboard nav (Tab key skips them — not focusable by default)
  • Screen readers (announced as "Source LinkedIn" instead of "Source LinkedIn, button, pressed")
  • Voice control ("Show numbers" / "Show grid" doesn't tag them as interactive)

Switching to <button type="button"> fixes all three for free. Real semantic HTML is almost always the right answer.

Focus management for modals

let _modalPrevFocus = null;
function openModal() {
  _modalPrevFocus = document.activeElement;
  $('modal-bg').classList.add('show');
  setTimeout(() => $('paste-area').focus(), 0);
}
function closeModal() {
  $('modal-bg').classList.remove('show');
  if (_modalPrevFocus && _modalPrevFocus.focus) _modalPrevFocus.focus();
}

When a modal opens: save where focus currently is, then move focus into the modal (so screen readers and keyboard users land there). When it closes: restore focus to where it was. Without this, keyboard users get dumped at the top of the page after dismissing a dialog. Universally expected behavior; trivially easy to miss.

11. visualViewport API — detecting the soft keyboard

if (window.visualViewport) {
  window.visualViewport.addEventListener('resize', () => {
    const kbOpen = window.innerHeight - window.visualViewport.height > 100;
    bar.style.display = kbOpen ? 'none' : '';
  });
}

What window.visualViewport is the part of the screen currently visible to the user — which shrinks when the soft keyboard pops up. window.innerHeight is the layout viewport (full document area). The difference between the two equals approximately the keyboard height.

Why we need it Our export-bar is position: fixed at the bottom on mobile. When the user taps the search input, the iOS keyboard appears — and the export-bar gets pushed up... over the search input itself. Hiding the bar while the keyboard is up fixes this.

When Any mobile-aware UI with bottom-fixed elements. Chat apps, search overlays, sticky CTAs.

12. :focus-visible — keyboard-only focus rings

.pill:focus-visible, .act:focus-visible {
  outline: 2px solid var(--accent);
  outline-offset: 2px;
}

The plain :focus pseudo-class fires on any focus — including mouse clicks, which is why developers historically wrote outline: none to hide the "ugly" focus ring after clicks. That broke keyboard users.

:focus-visible is browser-heuristic: it fires only when the browser thinks the user is navigating by keyboard (Tab key, arrow keys, etc.), not when they click. Now you can show a visible focus ring without it appearing on every click. It's the modern correct answer.

13. Concepts glossary

Same-origin policy browser security model
JavaScript can only fetch resources from the same scheme + host + port that served the page. github.io/Job_Scraper/triage.html can fetch github.io/Job_Scraper/jobs.json freely. Cross-origin fetches require CORS headers from the target server. This is why our fetch-at-view-time architecture works without any CORS dance — everything is same-origin.
CDN / edge cache distribution layer
A network of geo-distributed servers that cache content close to users. GitHub Pages uses Fastly. When your phone requests jobs.json, Fastly's nearest edge serves the cached copy unless the cache TTL has expired, then it re-fetches from origin (the GitHub repo).
IIFE — Immediately Invoked Function Expression JS pattern
(() => { ... })() — declare and call a function in one expression. Used to scope variables locally and return a value. Less common since ES6 modules and let/const blocks, but still useful for one-off initialization.
Optional chaining (?.) and nullish coalescing (??) ES2020 syntax
r.reason?.message evaluates to undefined if r.reason is null/undefined, instead of throwing. a ?? b returns b only if a is null or undefined (not just falsy). Together they make defensive coding terse.
Bubble phase / event delegation DOM event model
When you click an element, the event fires on the target, then on each ancestor up to document. A listener high up the tree can catch clicks from any descendant — that's delegation. e.target is the actual clicked element; e.currentTarget is the one with the listener.
localeCompare string comparison
"a".localeCompare("b") returns -1, 0, or 1 — usable directly as a sort comparator. Handles Unicode/locale-specific ordering (e.g. accented characters) correctly, unlike plain a < b which compares UTF-16 code units.
CSS Grid vs Flexbox layout systems
Grid is 2D (rows + columns simultaneously). Flexbox is 1D (one axis at a time). Use Grid for page-level layouts, KPIs, card grids. Use Flexbox for navigation bars, form rows, anything in a single line. They compose: a Grid cell can contain a Flex container, vice versa.
Reactive vs imperative rendering UI paradigms
React/Vue/etc. are reactive: you describe what the UI should look like given state, the framework figures out DOM mutations. Our dashboard is imperative: renderAll() explicitly rebuilds DOM via innerHTML. Imperative is simpler for tiny apps; reactive is essential past ~5 interactive components.

15. Next steps for deeper study

If this clicked and you want more

  1. Reactive frameworks: the imperative renderAll() pattern works for ~5 components; past that you want React, Vue, Svelte, or Solid. Start with Svelte if you want minimum ceremony, React if you want the largest ecosystem.
  2. TypeScript: the defensive guards we added (URL protocol check, null fallbacks) are the kind of thing TypeScript catches at compile time. Worth learning before your next non-trivial JS project.
  3. Service workers: turn the dashboard into a PWA — works offline, installable on iOS home screen, push notifications possible. web.dev/learn/pwa.
  4. Web Components: if you want to package the triage card / filter pill / chart as reusable elements, native Custom Elements + <template> are the no-framework path. Lit is the lightest abstraction over them.
  5. Observable / D3: if you ever want richer visualizations than the bar charts here, Observable is the playground and D3 is the deep tool. The job market visualization use case is a perfect fit.

The meta-lesson

The dashboard you have is ~700 lines of HTML/CSS/JS that does what would be a typical mid-sized React app. The reason it's small is that we picked patterns that compose:

  • Static hosting + fetch instead of a server
  • localStorage instead of a database
  • CSS custom properties instead of a CSS-in-JS theme system
  • Event delegation instead of N listeners
  • Real HTML buttons instead of a UI library

Each of these is "boring tech" — Web Platform features that ship in every browser. The lesson: lots of "JavaScript framework" features exist because the platform was insufficient at the time. The platform has caught up substantially. For small/personal projects, the framework-less path is often the simplest and most durable.
